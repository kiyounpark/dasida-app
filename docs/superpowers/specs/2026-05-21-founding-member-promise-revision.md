## 한 줄 요약

Founding Member 약속을 "6개월 전체 무료 + Founding에게 추가 1개월 무료"로 단순화한다.
구현 복잡도 0에 가까운 형태로 정리하고, 무료 기간은 "잃어버린 매출"이 아니라
"CAC 0원의 입소문 채널"로 재정의한다.

---

## 배경 및 문제 인식

`features/profile/components/founding-member-card.tsx`에 "앞으로 출시될 프리미엄 기능도
Founding Member에게는 1년간 무료" 문구가 박혀 있다. 그러나:

1. **표현 불일치**: spec(2026-04-14)은 "출시 후 12개월", code는 "프리미엄 출시 후 12개월"로
   기준점이 다르다. 둘은 다른 약속이다.
2. **시한폭탄**: code 해석으로 가면 프리미엄 출시 시점이 늦어질수록 Founding Member 혜택이
   무한정 미래로 밀린다.
3. **자기 의심의 출처**: 박기윤 본인이 "30% 할인 등 적용할 수 있을까", "유료결제 복잡해
   보인다"고 명시. 약속 자체보다 약속을 구현할 능력에 대한 불안이 컸다.
4. **심리적 한계**: 박기윤이 매출 없이 버틸 수 있는 한계는 "6개월"로 자기 진단. 따라서
   18개월짜리 약속(12+6 보너스)은 박기윤 사업의 현실과 맞지 않는다.

## 진단으로 얻은 핵심 통찰

### 통찰 1: 사용자 ≠ 결제자

- 사용자: 노력형 3등급 수험생
- 결제자: 부모
- 기존 monetization spec은 둘을 분리하지 않았다.

### 통찰 2: 부모는 학생을 통해서만 설득된다

- 부모는 다시다를 직접 평가하지 않는다.
- 결제 트리거 = "학생이 부모에게 '이거 꼭 필요해요'라고 부탁하는 순간"
- 따라서 부모용 별도 제품(리포트, 대시보드)을 초기에 만들 필요는 없다.
- 학생용 제품 안에 "부탁하고 싶어지는 가치"를 심는 것이 핵심.

### 통찰 3: Founding Member 무료 기간은 매출 차단이 아니라 입소문 인큐베이터

- Founding Member는 어차피 초기 100~300명 한정.
- 6개월 후엔 비-Founding 신규 유저가 결제 가능 인구의 다수를 차지한다.
- 6~7개월 무료는 "현금 대신 제품 접근권으로 지불하는 마케팅 비용", CAC = 0원.

### 통찰 4: 진짜 불안의 정체는 구현 능력에 대한 자기 의심

- 약속 문구를 다듬어도 구현 능력 불안은 안 사라진다.
- 따라서 약속은 **구현 복잡도가 가장 낮은 형태**로 가야 한다.

## 합의된 전제 (Premises)

1. 다시다의 사용자(학생)와 결제자(부모)는 다른 사람이다.
2. 부모는 학생을 통해 설득된다 — 학생용 제품에 "부탁하고 싶어지는 순간"을 심는다.
3. "학생이 부모에게 부탁하는 순간"이 핵심 결제 선행 지표다(7일 리텐션보다 직접적일 수 있음).
4. Founding Member 6~7개월 무료는 매출 차단이 아니라 입소문 마케팅 채널이다.
5. 인접 서비스(오르조 등)에서 "학생→부모 결제 부탁" 패턴은 관찰된다. 다시다에서는 미검증 —
   출시 후 직접 관찰해야 한다.

## 확정 약속 문구 (B-SIMPLE)

**6개월 카운트 방식: 전역 타이머 (App Store 정식 출시일 기준).**
모든 유저에게 동일한 종료 날짜를 적용한다. 유저별 가입일 기준이 아니다.

**UI 표시 원칙: 상대 기간이 아니라 절대 날짜로 표시한다.**
- ❌ "출시 후 6개월 무료"
- ✅ "2027년 X월 X일까지 무료" (출시일 확정 후 정확한 날짜 채움)

이렇게 하면 유저가 머릿속에서 계산할 필요가 없고, 늦게 가입한 유저도
가입 시점에 정확한 무료 마감일을 알게 되어 약속을 어긴 느낌이 없다.

**대내(spec) / 대외(앱 카피) 공통 문구 (출시일 확정 후 날짜 치환):**

> 다시다는 **YYYY년 MM월 DD일까지 모든 기능 무료**입니다.
> Founding Member에게는 **추가 1개월(YYYY년 MM월 DD일까지) 무료**를 드립니다.

### 약속 구조

| 구분 | 일반 유저 | Founding Member |
|------|---------|-----------------|
| 출시 ~ 출시일+180일 | 모든 기능 무료 | 모든 기능 무료 |
| 출시일+180일 ~ +210일 | 페이월 등장, 결제 또는 이탈 | 페이월 추가 30일 유예 |
| 출시일+210일 이후 | 결제 또는 이탈 | 결제 또는 이탈 |

**참고:** 페이월 만료 날짜는 출시일 확정 시점에 두 개의 상수(`PAYWALL_START_DATE`,
`FOUNDING_PAYWALL_START_DATE`)로 코드에 하드코딩한다. 유저별 가입일 추적은 Founding
Member 판별용으로만 필요하다.

### Founding Member 정의

- **"출시일" = App Store 정식 출시일** (TestFlight 베타 기간은 포함하지 않음).
  TestFlight 기간에 가입한 유저도 정식 출시 후 30일 안에 다시 로그인하면 Founding Member로 인정한다.
- 가입일이 **출시일 + 30일 이내**인 유저를 Founding Member로 본다.
- 클라이언트만으로 판별 가능(Firebase Auth metadata `creationTime` 또는 AsyncStorage 가입일).
- 서버 사이드 검증은 6개월 후 페이월 도입 시점에 함께 구현.

| 항목 | spec(04-14) | code 현재 | B-SIMPLE |

### Phase 1 — 출시 ~ 출시 후 3개월 (데이터 수집)

유료화 설계에 필요한 정량/정성 데이터를 수집한다. "6개월"이라는 시간 약속이 정당화되는
이유가 바로 이 수집 과정 때문이다.

**기존 인프라 발견:** `features/analytics/`에 GA4 Measurement Protocol + PostHog 이중
트래킹이 이미 구축되어 있고(`logEvent()` 한 함수가 양쪽으로 동시 전송), `event-types.ts`에
다음 이벤트들이 이미 정의되어 있다:

- `diagnosis_started`, `diagnosis_completed`
- `review_started`, `review_completed`
- `graduation_reached`
- `mock_exam_started`, `mock_exam_completed`
- `weakness_practice_started`, `weakness_practice_completed`
- `no_review_day_card_viewed`, `no_review_day_card_cta_pressed`
- `notification_opened`, `review_router_*`, `review_fallback_chat_completed`

페이월 위치 결정(어떤 기능 뒤에 둘 것인가)에 필요한 학습/복습 행동 이벤트는 이미 충분히
잡혀 있다. 추가로 박아야 할 이벤트는 두 개뿐이다.

**🔹 신규 추가 이벤트 2개 (출시 전 코드에 추가)**

| # | 이벤트 키 | 발생 시점 | 알려주는 것 |
|---|----------|---------|-----------|
| 1 | `signup` | 회원가입 완료 직후 | 가입자 수, 가입 속도, 채널별 유입 |
| 2 | `share_or_screenshot` | 공유 버튼 탭 또는 스크린샷 감지 | "학생→부모 부탁" 가설 1차 정량 신호 |

작업량: 각각 한 줄 (`logEvent('signup', { source })` / `logEvent('share_or_screenshot', { context })`),
`event-types.ts`에 타입 정의 추가 포함 30분 이내.

**🔹 리텐션은 PostHog Retention 대시보드 활용 (별도 이벤트 불필요)**

`signup`만 박으면 PostHog가 자동으로 cohort retention을 계산한다(7일, 14일, 30일).
별도 `app_open_d7` 이벤트를 박을 필요 없음. 코드 작업 0.

**🔹 사용자 인터뷰 모집: PostHog Surveys (코드 작업 0)**

박기윤이 우려한 "한 달 안에 반복 유저를 어떻게 연락할까" 문제의 해결책. PostHog가
이미 깔려 있으므로 SDK 추가 작업 없이 대시보드에서 Survey를 만들고 트리거 조건만
설정하면 된다.

**Survey 설정:**
- **트리거 조건**: 최근 7일 안에 `review_completed` 2회 이상 + 가입 후 10일 이상 경과
- **표시 방식**: 앱 내 작은 팝업/배너 (PostHog SDK가 자동 렌더링)
- **내용**: "다시다 더 좋아지게 도와주실 분, 30분 통화 인터뷰 응해주시면 기프티콘
  드릴게요. 이메일 남겨주세요."
- **수집 데이터**: 옵트인 이메일 → PostHog 대시보드에서 박기윤이 직접 명단 확인 후 연락

**왜 이 방식인가:**
- 박기윤이 일방적으로 유저 연락처를 캐는 게 아니라 자발적 옵트인 → 윤리적, GDPR 안전
- 응답률 높음 (이미 "통화 응함"에 동의한 사람들)
- 별도 백엔드 작업 불필요

**🔹 사용자 인터뷰 가이드 (옵트인된 사람 중 3명, 1:1 30분씩)**

PostHog Survey로 모인 명단 중 가장 활성도 높은 3명에게 직접 연락. 다음 5개 질문을 던진다.

1. 다시다를 처음 알게 된 경로는?
2. 사용 중 "오, 이거 도움 된다" 싶었던 구체적인 순간이 있었나?
3. 친구나 부모한테 다시다를 보여준 적 있나? 보여줬다면 반응은?
4. 만약 다시다가 내일 갑자기 사라진다면 그 빈자리를 뭐로 대체할 건가?
5. 만약 다시다가 유료라면 한 달에 얼마까지 낼 수 있겠나? (학생 본인 기준 / 부모가
   낸다면 다르게 답해도 됨)

**🔹 Founding Member 식별 유지**

이미 회원가입 시 `creationTime`이 Firebase Auth에 저장되므로 별도 작업 불필요.
페이월 노출 시점에 클라이언트에서 `creationTime < 출시일 + 30일`로 분기.

**🔹 수집 → 결정 흐름**

```
출시 ─→ 이벤트 누적 ─→ 30일차: PostHog Retention 대시보드 + Survey 옵트인 명단 확인
                          │
                          ├─ 7일 리텐션 30% 이상 ─→ 옵트인 명단 중 3명 1:1 인터뷰 ─→ Phase 2 페이월 설계
                          ├─ 10~30% ─→ 제품 개선 우선, 페이월 미룸
                          └─ 10% 미만 ─→ 핵심 가치 재검토 (이 경우 약속 날짜 연장 공지)
```

## Long-term Direction: "부모에게 요청하기" 결제 유도 기능 (별도 spec 예정)

본 spec은 Founding Member 약속 정정과 데이터 수집까지 다룬다. 그러나 박기윤이 추구하는
다시다의 진짜 결제 모델은 다음 한 줄로 요약된다.

> **학생이 앱 안에서 부모에게 직접 "이 앱 결제해주세요"를 한 번의 탭으로 요청할 수 있게 한다.**

토스의 "가족에게 돈 요청하기" 패턴과 유사한 흐름. 진단으로 도출한 통찰 1·2·3과 정확히
일치하는 제품 구조다:
- 사용자(학생)와 결제자(부모)가 다르다 → 둘을 한 화면 안에서 연결
- 부모는 학생을 통해 설득된다 → 학생이 직접 요청을 만든다
- 결제 트리거 = "학생이 부모에게 부탁하는 순간" → 그 순간을 앱이 만든다

**Phase 1에서 모으는 `share_or_screenshot` 이벤트는 이 기능의 사전 검증 신호**다.
유저가 자발적으로 다시다를 부모에게 보여주려 한다면, 명시적 "부모 요청" 버튼은 더
높은 전환을 만들 가능성이 크다.

**언제 별도 spec으로 분리되는가:** Phase 1 데이터 수집 후, 인터뷰에서 "부모에게
