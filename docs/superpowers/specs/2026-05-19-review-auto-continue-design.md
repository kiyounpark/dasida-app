# 복습 자동 이어풀기 (Auto-Continue) 설계

작성일: 2026-05-19
상태: 기획중

## 1. 배경 / 문제

현재 복습 흐름은 다음과 같다:

- 홈("학습 여정") 화면에 복습 카드 1장이 표시된다 (`nextReviewTask` 1개).
- 카드를 탭하면 복습 세션으로 진입한다.
- 복습 완료 시 `onComplete()`가 학습 기록 저장 / 다음 단계 task 생성 / 알림 재예약 / `refresh()`를 수행한 뒤 **`router.back()`으로 홈으로 돌아간다** (`features/quiz/hooks/use-review-session-screen.ts:604`).
- 오늘 복습이 여러 개여도 한 개 끝낼 때마다 홈으로 튕겨 나오고, 사용자가 카드를 **다시 탭해야** 다음 복습을 시작할 수 있다.

홈 hero 문구는 "오늘 복습 N개를 차례로 다시 볼 수 있어요"라며 연속 진행을 암시하지만, 실제로는 매번 홈을 거쳐 다시 탭해야 하므로 흐름이 끊긴다. 별도의 복습 전용 목록/허브 화면은 존재하지 않는다.

## 2. 목표

- 복습 하나를 끝내면 홈으로 돌아가지 않고 **자연스럽게 다음 due 복습으로 이어서** 진행한다.
- 복습과 복습 사이에 짧은 **징검다리(bridge) 화면**을 두어 완료감·진행률·휴식 지점을 제공한다.
- 사용자가 원할 때 **언제든 멈추고 홈으로 나갈 수 있다** (기존 뒤로가기 버튼 활용).
- 별도 복습 허브/목록 화면은 만들지 않는다 (YAGNI — 자동 이어풀기로 목록의 역할이 사라지므로).

비목표:

- 특정 복습만 골라 풀기 / 오늘 복습 목록 전체를 한눈에 보기 → 이번 범위 아님. 추후 사용자 요구 발생 시 재검토.
- 복습 진입점 변경 → 홈 카드 탭 진입은 현행 그대로 유지.

## 3. 사용자 흐름

오늘 복습 3개(분수 day3 / 소수 day1 / 비율 day7)가 있다고 가정한다.

1. 홈 복습 카드 탭 → 첫 복습(정렬상 가장 빠른 due) 세션 진입. **현행 유지.**
2. 분수 복습 완료 → **징검다리 화면** 표시:
   - "분수 복습 완료 ✅"
   - "오늘 복습 1 / 3 완료" (진행 표시 텍스트)
   - `[ 다음 → ]` 버튼 — 탭하면 즉시 다음 복습으로
   - 아무것도 안 하면 **2초 후 자동으로** 다음 복습으로 진행
   - 상단 앱바의 **기존 뒤로가기 버튼** = "오늘은 여기까지" → 홈 복귀 (이 시점에 직전 복습은 이미 저장 완료라 안전)
3. 소수 복습 세션 자동 진입 → 완료 → 징검다리("2 / 3 완료") → 비율 복습 자동 진입
4. 비율(마지막) 복습 완료 → **전체 완료 화면**:
   - "오늘 복습 다 끝냈어요 🎉"
   - "3개 전부 복습 완료"
   - 홈으로 돌아가는 버튼
5. 중간에 뒤로가기로 빠져나가면 → 그때까지 완료한 복습은 저장된 상태로 홈 복귀. 다음에 다시 들어오면 남은 due 복습부터 이어진다.

## 4. 화면 / 컴포넌트 설계

### 4.1 재활용 (신규 디자인 없음)

- **복습 세션 화면** (`review-session-screen-view.tsx`) — 구조 그대로. 같은 앱바("오늘의 복습", 뒤로가기) 유지.
- **전체 완료 화면** — 기존 `DoneView` (`features/quiz/components/review-session/done-view.tsx`)의 졸업/완료 톤(스탬프, Paper 토큰, 색상)을 재활용하고 문구만 "오늘 복습 다 끝냈어요 / N개 전부 복습 완료"로 변형한다.

### 4.2 신규 — 징검다리 컴포넌트 1개

- `DoneView`의 디자인 언어를 축소 변형한 **bridge 뷰**.
- 표시 요소:
  - 완료 표시 (직전 복습 약점 라벨 + ✅)
  - 진행 텍스트: `오늘 복습 {완료수} / {전체수} 완료` (필수)
  - `[ 다음 → ]` 버튼
  - 2초 자동 진행 타이머
  - 점 인디케이터(●●○)는 **넣지 않는다** (복습 10개+ 시 지저분). 텍스트 진행 표시만.
- 같은 화면(`review-session-screen-view`)의 앱바를 그대로 쓰므로 뒤로가기 버튼이 자연스러운 탈출구가 된다.

## 5. 동작 / 로직 설계

### 5.1 "다음 due 복습"을 물고 들어가기

현재 `onComplete()`는 마지막에 `router.back()`을 호출한다. 이를 다음과 같이 변경한다:

- `completeReviewTask` / `spawnMistakeReviewTasks` / `refresh()`까지는 현행 유지.
- 그 후 **남은 due 복습**을 판정한다:
  - `refresh()` 후의 최신 상태에서 `dueReviewTasks`(오늘 기한 이하 미완료, `scheduledFor` 오름차순)를 읽는다 — 판정 로직은 기존 `local-learning-history-repository.ts`의 `buildReviewTaskState`를 재사용/참조.
  - **주의:** 방금 완료한 task가 만든 다음 단계 task(예: day3)는 미래 날짜이므로 `dueReviewTasks`에 안 잡혀야 정상이다. 또한 `spawnMistakeReviewTasks`가 만든 내일(day1) task도 due 아님. → 오늘 새로 due가 되는 task만 이어진다.
- 남은 due 복습이 있으면 → 징검다리 상태로 전환 (홈으로 안 나감).
- 남은 due 복습이 없으면 → 전체 완료 화면 표시 후 사용자가 버튼으로 홈 복귀.

### 5.2 징검다리 → 다음 복습 전환

- 징검다리에서 "다음" 또는 2초 타이머 만료 시:
  - 다음 due task로 세션 상태를 **리셋**한다 (현재 task/steps/각종 ref/`sessionComplete` 초기화 후 다음 taskId로 재로드).
  - 구현 방식 후보 (구현 계획 단계에서 확정):
    - (a) 같은 라우트에 `router.replace`로 다음 taskId 파라미터 전달 → 화면 자연 리마운트로 상태 초기화. 뒤로가기 스택이 쌓이지 않아 깔끔.
    - (b) 훅 내부 상태 머신으로 task 큐를 들고 in-place 리셋.
  - **권장: (a) `router.replace`** — 기존 세션 초기화 로직(ref 다수)을 재현하는 부담이 적고, 뒤로가기 한 번에 홈으로 나가는 동작이 자연스럽다.

### 5.3 진행률 계산

- 분모(전체): 이 이어풀기 세션을 **시작한 시점**의 오늘 due 복습 개수. (도중에 오답으로 새 day1 task가 생겨도 분모를 늘리지 않는다 — 오늘 due가 아니므로 흐름에도 안 들어오고 카운트도 안 흔들린다.)
- 분자(완료): 이번 이어풀기 세션에서 완료한 복습 수.
- `router.replace` 방식을 쓸 경우 시작 시점 분모가 리마운트로 사라지므로, 분모/분자를 라우트 파라미터로 넘기거나 세션 스코프 저장소에 보관한다 (구현 계획에서 확정).

### 5.4 탈출 (오늘은 여기까지)

- 별도 UI 추가 없음. 복습 세션/징검다리 화면 상단 앱바의 **기존 뒤로가기 버튼**(`onBack` → `router.back`)을 그대로 사용.
- 뒤로가기 시점엔 직전 완료 복습이 이미 저장된 상태이므로 데이터 손실 없음.
- 진행 중이던(아직 완료 안 한) 복습 세션에서 뒤로가기: 현행 동작과 동일하게 미완료 처리(저장 안 됨).

## 6. 엣지 케이스

- **복습 1개뿐**: 완료 → 징검다리 없이 바로 전체 완료 화면 (남은 due 0). 또는 단일 케이스에서는 기존 단일 완료 화면 유지 — 구현 계획에서 "1개일 때 전체완료 화면 문구"를 확정.
- **마지막 복습 완료**: 남은 due 0 → 전체 완료 화면.
- **day30 졸업 복습이 섞여 있음**: 졸업 복습도 하나의 due task로 동일하게 흐름에 포함. 징검다리/전체완료 문구는 기존 졸업 톤과 충돌하지 않게 처리(졸업은 개별 완료 시 done-view의 졸업 표시 유지 여부 구현 계획에서 확정).
- **오답으로 새 day1 task 생성**: 오늘 due 아니므로 이번 이어풀기에 포함되지 않음. 분모 불변.
- **2초 자동 진행과 사용자 뒤로가기 경합**: 뒤로가기가 우선. 화면 이탈 시 자동 진행 타이머는 반드시 정리(clear)한다.
- **앱 백그라운드 → 복귀**: 징검다리에서 백그라운드 갔다 오면 타이머 상태 정의 필요 — 단순화를 위해 "포커스 잃으면 타이머 정지, 사용자 탭으로만 진행" 정도로 둘지 구현 계획에서 확정.

## 7. 구현 성격 요약

- 새 라우트/허브 화면: **없음**
- 변경: `use-review-session-screen.ts`의 `onComplete` — `router.back()` 대신 "남은 due 판정 → 징검다리 / 전체완료" 분기
- 신규 컴포넌트: 징검다리 뷰 1개 (`done-view` 변형)
- 변형: 전체 완료 화면 (기존 `done-view` 톤 재활용, 문구만 변경)
- Claude 디자인: **불필요** (기존 디자인 시스템 내 구현)

## 8. 검증 / 테스트 관점 (Expo/RN)

- 복습 2개+ 시 완료 → 징검다리 → 자동 2초 진행 → 다음 세션 정상 리마운트 / 상태 초기화 확인.
- "다음" 즉시 탭 시 타이머와 중복 진행 안 되는지.
- 징검다리에서 뒤로가기 → 홈 복귀 + 직전 복습 저장 확인.
- 마지막 복습 후 전체 완료 화면 노출 + 홈 복귀.
- 자동 진행 타이머가 화면 이탈/언마운트 시 정리되는지 (메모리/네비게이션 누수 없음).
- 복습 1개뿐일 때 흐름.
- 기존 단위 테스트(`use-review-session-screen.test.ts`, `e2e/review-session.spec.ts`)에 미치는 영향 점검 및 케이스 추가.
