# 사진 flow — 풀이 자동 인식 라우팅 (질문 생략) 설계

> 날짜: 2026-07-22
> 상태: 기획중 (기윤과 논의 중)
> 상위 문서: 노션 "📤 PDF/사진 업로드 오답노트 대행 (프로토타입)" (2026-07-20~21 결정 포함)

## 한 줄

사진에 이미 학생의 풀이가 찍혀 있으니, **"어떤 방법으로 풀었나요?" 질문을 없애고 AI가 풀이를 읽어 스스로 파악** → 기존 진단 flow에 자동 입장한다.

---

## 1. 배경 — 지금은 학생 입으로 받는다

현재 텍스트 기반 flow (기출 모의고사 진단):

```
"어떤 방법으로 풀었나요?"           ← use-exam-diagnosis.ts:166
  → 학생이 타이핑 (rawText)
  → analyzeDiagnosisMethod(rawText)  ← diagnosis-router.ts (OpenAI 라우터 + mock 폴백)
  → confidence ≥ 0.74 → flow 자동 입장
  → 애매하면 needsManualSelection → 후보 카드에서 학생이 고름
```

사진 flow에서는 학생이 풀이를 이미 종이에 써놨다. 그걸 또 말로 물어보는 건 이중 입력 = 귀찮음 축 위반.

## 2. 핵심 결정 — vision 한 번에 methodId까지 뽑는다

사진 분석 단계(어차피 답 읽으려고 vision 호출함)에서 **한 번의 호출로** 뽑는다:

```
vision 출력 (문제당): {
  답: 3번,
  풀이 전사: "판별식 쓰고 근의 공식 대입…",
  predictedMethodId: 'quadratic',    ← 기존 SolveMethodId 그대로
  confidence: 0.9,
  candidateMethodIds: ['quadratic', 'factoring'],
}
```

- 이 출력을 기존 `DiagnosisRouterResult` 모양에 맞춰 넣는다.
- **뒷단(0.74 임계값 · 후보 카드 · flow 엔진)은 무수정.** "현관은 새로, 착륙장은 그대로" 원칙 그대로.
- 프롬프트 재료: `data/diagnosis-method-routing.ts`의 카탈로그(labelKo·summary·keywords·exampleUtterances)를 그대로 vision 프롬프트에 붙인다.

### 왜 "전사 → 텍스트 라우터" 2단이 아니라 vision 직행 1단인가

- 이미지에는 전사보다 신호가 많다 (그림·화살표·지운 흔적).
- 호출 1번 = 비용·지연 절반.
- 전사는 버리지 않고 같이 받아둔다 (근거 제시 + 디버깅용).

## 3. UX — 질문 삭제가 아니라 단언으로 교체

> ~~"어떤 방법으로 풀었나요?"~~
> → **"풀이 보니까 근의 공식으로 접근했네. 그럼 여기서부터 보자."**

앱이 **네 손글씨를 읽었다**는 걸 보여주는 순간 자체가 와우 모먼트. "찍은 것 = 틀린 것"과 같은 결 — 학생이 입력할 것을 앱이 알아챈다.

### 갈래 3개

| 상황 | 동작 |
|------|------|
| 풀이 흔적 있음 + 확신 높음 | 단언 + flow 자동 입장 (질문 생략) |
| 풀이 흔적 있음 + 애매 | 기존 후보 카드 — 허용된 "딱 하나의 되물음" |
| 풀이 흔적 없음 (암산·지움) | 기존 질문("어떤 방법으로 풀었나요?")으로 폴백 |

AI가 우기는 구간 없음 — 7/21 자신감 게이트 원칙과 동형.

## 4. 검증 — 검증 A에 그대로 얹는다

기존 검증 A = 읽기 정확도 + 약점 라우팅 착지 정확도.
여기에 **"풀이 사진 → methodId 분류 정확도"가 착지 정확도 항목으로 포함**된다.

- 방법: 기윤이 직접 문제 풀어 사진 제작 (7/22 계획 그대로) → GPT-4.1 손 검증
- 같이 볼 것: confidence가 정직한가 (애매한 풀이에 낮게 나오는가) — 커트라인 튜닝 겸용

### 모델 변경 (2026.07.22 결정)

- 7/20 "Sonnet으로 시작" → **GPT-4.1로 시작**으로 변경.
- 이유: 비용 차이는 이 규모에서 무의미(사진 1장 10~30원). 진짜 요인은 세팅 마찰 — OpenAI 키·호출 코드가 백엔드에 이미 있어 오늘 바로 검증 가능. 모델은 설정값이라 정확도 부족하면 그때 Claude로 승격 (데이터 기반 전환).

## 5. 논의할 것 (미결)

1. **단언 뒤 탈출구**: "풀이 보니 ~네" 단언에 "아니야" 버튼을 줄까?
   - 주면: 오판 안전망. 안 주면: 마찰 최소 + 오판 시 flow가 안 맞아 어차피 드러남.
2. **커트라인 재사용**: 텍스트 라우터의 0.74를 그대로 쓸까, vision용 별도 값을 둘까? (실측 후 결정 — 책상에서 못 정함)
3. **풀이 전사 노출**: 학생에게 "내가 읽은 네 풀이"를 보여줄까, 내부용으로만 둘까?
4. **"풀이 흔적 없음" 판정**: vision이 스스로 판정하게 할까 (예: transcription 빈 값 + confidence 0), 별도 필드를 둘까?
