# 웹 1문제 진단 페이지 (트랙 B v1) — Design Spec

- 날짜: 2026-07-10
- 상태: 기획 (구현 전)
- 배경: 오르비 첫 커뮤 글 발행(2026-07-10) 이후 트랙 B — "다시다 기능 한 조각을 웹으로 떼서 무료 제공" — 의 첫 조각 확정.

## 1. 목적

다시다의 핵심 차별점인 **"풀 줄 아는 문제를 왜 또 틀리는지 = 풀이 어느 단계에서 어긋났는지"** 진단 경험을,
앱 설치·회원가입·데이터 입력 없이 **웹 URL 하나로 1분 안에** 체험하게 한다.

- 타겟: 실속파 수험생, 그중 "필사는 비효율이라 접었는데 '왜 틀렸나'까지 같이 버린" 상위권 (②-b)
- 성격: 제품이 아니라 **부족에게 건네는 명함**. "그것만 써도 이득"인 최소 도구.
- 커뮤니티(오르비 등) 원칙과 결: 글 본문에는 앱 언급 0개 → 물어보면 이 URL을 꺼낸다.

## 2. 핵심 경험 (확정 사항)

1. **문제는 단 1개.** 유명하고 난이도가 적절한 실제 기출.
   - v1 기본 후보: 2027학년도 6월 모평 수학 공통 21번 (앱에 약점 태깅 완료된 문제. 최종 선정은 구현 시작 시 기윤이 확정하되, 이 스펙의 구조는 문제가 바뀌어도 동일)
2. **방문자는 문제를 직접 푼다.** 모바일 웹에서 문제 이미지를 보고 답을 입력 (21번 기준 단답 숫자 입력).
3. **틀린 경우:** "어떻게 풀었는지"를 선택지로 고르면 → **풀이의 어느 단계에서 어긋났는지** 짚어준다.
4. **맞은 경우:** 축하 + **"이 문제에서 사람들이 가장 많이 무너지는 단계"** 지도를 공개한다.
   - 맞아도 틀려도 최종 화면은 같은 "무너지는 지점 지도" — 출구 하나, 만드는 것도 하나.
5. **정직 원칙:** v1에는 실제 방문자 통계가 없으므로, "무너지는 단계" 데이터의 출처를 화면에 명시한다
   (예: "다시다 약점 태깅·해설 분석 기준"). 없는 통계를 있는 것처럼 보이게 하지 않는다.
6. **마무리 CTA는 가볍게 1개:** "네 오답에도 이걸 하고 싶다면 → 다시다" (스토어 링크). 강요 없음.

## 3. 접근 방식 (선택지 비교)

| 접근 | 내용 | 판정 |
|------|------|------|
| **A. 초경량 단일 정적 페이지** | 한 페이지에 문제·입력·진단·지도. 데이터는 하드코딩. 서버·로그인 없음 | ✅ 채택 |
| B. Expo 웹 빌드로 앱에서 떼기 | 기존 앱 컴포넌트를 웹으로 내보내기 | ❌ 1문제에 과함. 네이티브 의존성 리스크 |
| C. EAS Hosting + API 라우트 | 서버에서 문제/통계 관리 | ❌ v1 데이터가 1문제뿐 (YAGNI) |

**A 채택 이유:** 명함은 가볍고 빨리 만들어 돌리는 게 이긴다. 반응이 오면 그때 B/C로 키운다 (출시 = 세이브포인트).

## 4. 화면 흐름 (단일 페이지, 상태 4개)

```
[1. 문제]        문제 이미지 + 한 줄 훅 + 답 입력(숫자) + 제출
     │
     ├─ 오답 → [2. 풀이 선택]  "어떻게 풀었어요?" 접근 방식 선택지 (3~5개, 태깅 기반)
     │              │
     │              ▼
     │         [3. 진단]      "여기서 어긋났어요" — 어긋난 단계 하이라이트 + 짧은 설명
     │
     └─ 정답 → [3'. 지도]     "맞았어요. 근데 사람들은 여기서 무너져요" — 같은 단계 지도
                    │
                    ▼
               [4. 마무리]     단계 지도 전체 + 출처 명시 + CTA 1개 (다시다 링크)
```

- 상태 [3]과 [3']은 같은 "단계 지도" 컴포넌트를 쓰고, 하이라이트 대상만 다르다.
- 전체 소요 목표: **1분 이내** (문제 풀이 시간 제외).

## 5. 데이터 (하드코딩, 파일 1개)

문제 1개당 JSON/TS 객체 하나:

```
problem: { id, 출처 라벨, 문제 이미지, 정답(숫자), 한 줄 훅 }
steps:   [ { 단계명, 설명 } ... ]            // 풀이 단계 지도 (앱 약점 태깅에서 추출)
approaches: [ { 선택지 문구, 어긋난 단계 index, 진단 코멘트 } ... ]  // 오답자용
commonBreak: { 단계 index, 코멘트, 출처 문구 }   // "가장 많이 무너지는 단계"
```

- 출처: 앱에 이미 있는 6월 모평 수학 약점 태깅 + 해설 데이터를 사람이 옮겨 적는다 (자동 파이프라인 없음).
- 문제 이미지: 앱 자산의 해당 문제 크롭 이미지 재사용.

## 6. 기술 구성

- **형태:** 정적 단일 페이지. **위치는 앱 저장소 안 `web/` 폴더** (독립 저장소 아님 — 앱 자산 복사가 쉽고 빨리 시작). 앱 코드와 빌드 결합 없음.
- **스택:** 정적 HTML/CSS/Vanilla JS. 프레임워크·번들러 없음.
- **호스팅:** **Netlify** 정적 호스팅 무료 티어. `netlify.toml`의 base를 `web`으로 지정해 앱은 빼고 web만 배포. 커스텀 도메인은 v1 범위 밖.
- **모바일 우선:** 오르비/커뮤 유입은 대부분 모바일. 세로 화면 기준으로 디자인, 데스크톱은 가운데 정렬로 충분.
- **서버·로그인 없음.** 
- **측정(GA4):** **순수 GA4**로 행동 데이터를 수집한다 — analytics.google.com에서 웹 데이터 스트림(URL)만 등록하고 `gtag` 스니펫 한 줄을 붙인다 (Firebase SDK·웹앱 등록 불필요, 앱과 별개 속성). 트랙 B는 "실험"이므로 측정이 필수. 수집 데이터는 **기윤이 대시보드에서만** 보고, 유저 화면에는 표시하지 않는다 (화면의 정직 문구 유지). 이벤트: `page_view`(자동)·`problem_submit`·`result`·`approach_select`·`cta_open`/`store_click`.

## 7. 범위 밖 (v1에서 하지 않는 것)

- 문제 2개 이상 / 과목 세트
- **수집한 통계를 화면에 표시** (예: "남들이 무너지는 단계"를 실제 GA 데이터로 실시간 반영 → v2. v1은 다시다 분석 기준 문구 고정)
- 회원가입, 오답 저장, 앱 연동 딥링크
- Expo 웹 빌드, 앱 코드 재사용

## 8. 성공 기준

- 모바일 웹(사파리/크롬)에서: 문제 보고 → 답 입력 → 진단/지도까지 **끊김 없이 1분 내** 도달
- 오르비에서 "뭐 쓰냐"고 물었을 때 **링크 하나로 답이 되는 상태**
- 기윤 본인이 보고 "이거 하나만 써도 이득"이라고 납득하면 발행 (완벽선이 아니라 충분선)

## 9. 테스트 / 검증

- 수동 검증: 실기기 모바일(iOS 사파리, 안드 크롬)에서 4개 상태 전부 통과
- 정답/오답/무입력 각각 확인, 진단 코멘트와 태깅 원본 대조 (해설 검증은 앱 데이터 기준과 동일하게)
- 자동 테스트는 v1 범위 밖 (정적 페이지 + 하드코딩 데이터)
