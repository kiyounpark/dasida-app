## Problem Statement

복습 세션 마지막(`DoneView`)에서 사용자가 `🤔 다시 볼게요` / `✓ 기억났어요!` 두 버튼 중
하나를 눌러 다음 복습 일정을 스스로 정한다. 이 자기평가 방식에 세 가지 구조적 결함이 있다.

1. **묻는 타이밍이 편향됨.** 설명·힌트·요약을 모두 본 직후에 "기억났어요?"를 물으므로,
   재인(recognition) 상태에서 거의 항상 "기억났어요"를 누른다. 인출(retrieval) 능력을
   재지 못한다.
2. **좋은 신호를 안 씀.** 세션 중 답을 보기 전 첫 시도 정답 여부(`firstAttemptCorrectRef`)와
   고른 오답이 드러내는 약점(`Choice.weaknessId`)이 이미 기록되는데, 스케쥴링은 이를
   무시하고 사후 버튼 하나로 스테이지 승급을 결정한다.
3. **한 버튼이 두 축을 섞음.** "기억났나(실력)"와 "또 보고 싶나(의지)"가 한 버튼에
   합쳐져, 복습 일정이 실력이 아니라 그날의 선택에 좌우된다.

## What Makes This Cool

사용자는 "완료"만 보고 끝낸다. 뒤에서 앱이 "무엇을 틀렸는지"를 보고 그 **상세 약점**만
골라 다음 복습으로 자동 편성한다. 자기평가 버튼이 하던 일을 실제 성적이 정직하게 대체한다.
"앱이 내가 뭘 틀렸는지 알고 그것만 다시 챙긴다"는 경험이 학습앱의 핵심 신뢰를 만든다.

## Premises (사용자 승인 완료 2026-05-17)

1. 복습 완료 화면 = **"완료"만**. `다시 볼게요` / `기억났어요` 버튼 삭제.
   day30 졸업 연출(`isGraduated`)은 유지.
2. 세션 종료 시, **첫 시도 오답 스텝**(`firstAttemptCorrectRef[i] === false`)에서
   그 실수가 데려간 보충 경로의 약점 전부 = `discoveredPerStepRef[i]`를 모아
   복습 task를 생성/갱신한다. (목표: "완전정복" — 새는 구멍 없음. B는 오답에 따라
   갈라지는 개인화 경로라 노이즈가 아니라 진짜 신호임을 코드로 확인함.)
   "모든 오답 선택지는 `weaknessId` 필수" 불변식 테스트를 추가하고 누락분
   **6개**(전부 `basic_concept_needed` step1~3, choice c0·c2)를 채운다.
3. 상세 약점에 이미 상위 단계(day3/day7/day30) 복습 task가 있고 또 틀렸으면 →
   해당 task를 **삭제하고 `__day1` task를 신규 생성**(내일 예약). stage 필드만
   바꾸면 id(`sourceId__weaknessId__stage`)와 불일치하므로 in-place 변경 금지.
   day1 task가 이미 있으면 `scheduledFor`만 내일로 갱신. 약점당 미완료 task 1개 유지.
4. 오늘 본 큰 약점 task는 완료 시 사다리(day1→day3→day7→day30→졸업)를 **정상 진행**한다.
   틀린 세부 약점은 전제 2·3이 별도 관리하므로 시스템은 정직하게 유지된다.

## Constraints

- 데이터 모델 변경 없음. `ReviewTask` id = `sourceId__weaknessId__stage` 유지.
- `Choice.weaknessId`, `firstAttemptCorrectRef`, `discoveredPerStepRef`는 이미 존재 —
  새로운 데이터 수집 불필요.
- task는 약점당 1개 유지 → 무한 증식 없음.
- 알림 재예약(`rescheduleAllReviewNotifications`)은 기존 흐름 재사용.

## Approaches Considered

### Approach A: 최소 변경 — 완료 핸들러 단일화 + 오답 스폰 (권장)
- `DoneView`에서 `onPressRetry` 경로/버튼 제거. 비졸업도 단일 `[완료]` 버튼.
- `onPressRetry` 및 관련 wiring 삭제. `onPressRemember`를 `onComplete`로 일원화.
- `onComplete` 내부: 기존 `recordAttempt` + `completeReviewTask`(전제 4) 후,
  새 함수 `spawnMistakeReviewTasks(...)` 호출(전제 2·3).
- `review-content-map.test.ts`에 "모든 오답 weaknessId 필수" 테스트 추가 + 누락분 태깅.
- Effort: M / Risk: Low
- Reuses: `recordAttempt`, `completeReviewTask`, `createTaskId` 패턴, `alreadyExists`
  가드, `rescheduleAllReviewNotifications`.

### Approach B: 스케쥴러에 통합
- `spawnMistakeReviewTasks` 로직을 `review-scheduler.ts`의 `completeReviewTask`
  내부로 흡수, 세션 hook은 오답 약점 목록만 전달.
- Effort: M / Risk: Med (scheduler 책임 비대, 기존 호출처 영향 검토 필요)
- Reuses: 동일. 단 scheduler 단위 테스트 갱신 폭이 큼.

## Recommended Approach

**Approach A.** `spawnMistakeReviewTasks`는 `features/learning`의 순수 함수로 두고
(scheduler 헬퍼 `addDaysToToday`/`createTaskId` 재사용), 세션 hook은 `onComplete`에서
순서대로 호출만 한다(F5). 단 기존 `alreadyExists`(전체 id=stage 포함 비교)는
중복키로 부적합 — `(sourceId, weaknessId)` stage-무시 조회를 새로 구현한다.

## Spawn 규칙 상세 (전제 2·3 구체화 + 검증 반영)

**데이터 소스 (F2 확정):** 입력은 `firstAttemptCorrectRef`(어느 스텝을 틀렸나) +
`discoveredPerStepRef`(그 스텝의 오답이 데려간 보충 경로의 약점 전부)뿐.
`firstAttemptChoiceIndexRef` 단일 선택지로 좁히지 않는다(완전정복 목표).

1. `firstAttemptCorrectRef[i] === false` 인 step `i`만 수집.
2. 그 step들의 `discoveredPerStepRef[i]`를 모두 합쳐 `Set<WeaknessId>` 구성
   (세션 내 종류별 1회로 1차 중복 제거).
3. **중복 제거 키 = `(sourceId, weaknessId)` — stage 무시.** 각 약점 `w`에 대해
   미완료(`completed === false`) task를 (sourceId, w)로 조회(stage 불문):
   - 없음 → `__day1` task 신규 생성, `scheduledFor = 내일`.
   - 있음 & `stage !== day1` → **그 task 삭제 후** `__day1` task 신규 생성
     (`scheduledFor = 내일`). stage 필드 in-place 변경 금지(F3: id/stage 불일치 방지).
   - 있음 & `stage === day1` → `scheduledFor`만 내일로 갱신(신규 생성 안 함).
4. 완료 후 `rescheduleAllReviewNotifications` 1회.

**실행 순서 (F4 확정):** `onComplete`는 반드시
`recordAttempt` → `completeReviewTask`(store 저장) → **store 재로드** →
`spawnMistakeReviewTasks` → `rescheduleAllReviewNotifications` → `router.back()`
순서. 큰 약점이 곧 틀린 상세 약점이면, `completeReviewTask`가 방금 만든
다음 단계 task를 spawn이 재로드 후 인식해 day1로 교체해야 한다(전제 3 > 전제 4).

**배치 (F5):** `spawnMistakeReviewTasks`는 `features/learning`(예: `review-scheduler.ts`
옆 또는 동일 파일)의 순수 함수로 둔다. hook은 호출만(구조 규칙: 순수 계산은 domain).

엣지케이스:
- 모든 step 정답 → 보충 미진입 → `discoveredPerStepRef` 비어 spawn 없음.
- 상세 약점 == 세션 큰 약점 → 3번 규칙(삭제+day1 재생성)이 적용. 순서는 F4.
- `weaknessId` 미태깅 오답 → 불변식 테스트로 사전 차단 + 보충 경로 약점으로도 포착됨.

## Success Criteria

- 비졸업 복습 완료 화면에 자기평가 버튼이 없다(완료 1버튼).
- 첫 시도 오답이 있던 세션 종료 후, 보충 경로 약점들의 pending task가
  `(sourceId, weaknessId)` 기준 중복 없이 day1·내일로 생성/갱신된다
  (기존 상위 단계 task는 삭제 후 `__day1` 재생성).
- 같은 약점이 한 세션에서 여러 번 스쳐도 미완료 task는 최대 1개.
- 모든 step 정답 세션은 spawn 0, 큰 약점 사다리만 진행.
- 상세 약점 == 큰 약점일 때, `completeReviewTask` 직후에도 해당 약점 미완료
  task가 2개가 되지 않는다(F4 순서/재로드 회귀 테스트).
- `review-content-map.test.ts`: 모든 `correct:false` 선택지에 `weaknessId` 존재 — 통과
  (누락 6개 태깅 후).
- 기존 `onPressRetry` 참조 0건(삭제 검증).

## What I noticed about how you think

- "그냥 그날 복습을 완료하면 완료만 있으면 좋겠고" — UX를 빼는 결정을 먼저 했다.
  기능 추가가 아니라 신뢰할 수 없는 입력을 제거하는 방향을 직접 골랐다.
- "큰 약점이 아니라 상세한 약점을 기록하기로 하지 않았나요 2번에서 좀 틀린 게 있는 거
  같은데요" — 합의된 원칙과 내가 끼워넣은 폴백 사이의 모순을 직접 잡아냈다.
  이 지적이 "폴백을 잘 만들자"에서 "폴백이 필요 없게 데이터 구멍을 막자"로
  설계를 한 단계 끌어올렸다.
- 어려운 설명을 만나면 "쉽게 풀어달라"고 멈춰 세웠다. 이해하지 않은 채 진행하지 않았다.
